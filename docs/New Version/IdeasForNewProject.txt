Ideas for New Project (from current system review)
==================================================

1) Current implementation status
- The project is already partially hybrid:
  - BT orchestrates high-level phases.
  - Intake uses rule-based planning.
  - Runtime uses dynamic state-driven planning after handoff.
- It is not yet fine-grained at execution level:
  - Many physical actions run inside coarse workflow functions.
  - A single failure usually propagates to workflow failure.
  - Operator recovery choices are not first-class execution states yet.

2) Why this matters
- Real lab failures (misalignment, decap issues, temporary station faults) are common.
- Without structured recovery:
  - Throughput drops (full abort/restart).
  - Resume behavior is weak.
  - Operator interventions are not represented consistently in world state.

3) Recommended architecture if started from scratch
- Hierarchical fine-grained BT (not fully flat, not monolithic).
- Reusable action subtrees:
  - Navigate+LandmarkScan
  - Pick/Place sample
  - Pick/Place rack
  - ProcessAt3FingerStation (determine/decap/cap)
  - Device actions (centrifuge open/start/close/move)
- Recovery-aware wrapper around physical action nodes:
  - Retry
  - Skip action
  - Skip sample
  - Manual outcome patch
  - Abort
- Event-sourced world updates + deterministic resume.
- Planner/executor separation:
  - Planner proposes next intent from current world.
  - Executor performs intent and handles recovery.

4) Recommended strategy for the current project
- Do not full-rewrite now.
- Do incremental rework:
  1. Keep existing world model + planning logic.
  2. Add recovery policy and recovery-aware execution wrapper.
  3. Start with high-failure operations:
     - 3FG processing (especially decap)
     - Sample pick/place transfers
     - Rack transfer in/out of devices
  4. Gradually split coarse phase logic into explicit reusable leaf actions.

5) Human-in-the-loop recovery model (target behavior)
- On recoverable failure:
  1. Action node enters WAITING_RECOVERY (returns RUNNING).
  2. UI shows allowed choices for that context.
  3. Operator confirms what was physically done.
  4. World is patched atomically from chosen outcome.
  5. Execution continues with retry/skip/replan/abort.
- This should be sample-local when possible (avoid global freeze).

6) Concrete decap recovery outcomes (example)
- Outcome A: sample returned capped to centrifuge rack
  - location = centrifuge rack slot
  - cap_state = CAPPED
  - DECAP not completed
- Outcome B: sample moved capped to IH-500 rack
  - location = IH-500 slot
  - cap_state = CAPPED
  - DECAP not completed
- Outcome C: sample decapped and moved to IH-500 rack
  - location = IH-500 slot
  - cap_state = DECAPPED
  - DECAP completed

7) Where future changes are expected
- workflows/ (execution orchestration + recovery handling)
- planning/ (skip/replan semantics + process policy alignment)
- engine/ (BT recovery wrapper + command failure contract)
- integration/planner_web_interface/ (decision command channel + resume handling)
- world/ (manual outcome patch helpers + validation)
- Library/ API helpers (structured recovery request/ack handling)

8) Design principle
- Keep recovery policy external and data-driven.
- Keep world updates explicit, auditable, and deterministic.
- Replan from world truth after any operator intervention.

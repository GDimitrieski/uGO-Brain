IH-500 Device Integration Gaps
==============================

Context:
- IMMUNOANALYSIS execution is currently implemented in workflow logic via rack transfer steps.
- Task catalog currently defines SingleDeviceAction ACT values [1, 2, 3, 4, 30] with centrifuge-oriented semantics.

Known gaps:
1) No dedicated IH-500 runtime device interface/class (PackML/service wrapper) exists in Device/.
2) No IH-500-specific SingleDeviceAction ACT mapping is defined in Available_Tasks.json.
   - Current ACT map is: OpenHatch, StartCentrifuge, CloseHatch, MoveRotor, ScanLandMark.
3) IMMUNOANALYSIS execution currently performs rack push-in / pull-out only.
   - No explicit analyzer command sequence: start analysis, monitor running state, wait for completion.
4) No analysis result acquisition/validation from IH-500 is integrated into world/LIS flow.
5) Process completion is marked after unload by workflow logic, not by confirmed IH-500 completion event/result.
6) IH-500 cycle currently expects strict rack-index symmetry between:
   - uLMPlateStation JIG_ID=12
   - BioRadIH500Station JIG_ID=50
   Partial/mixed rack distributions are rejected.
7) No IH-500 fault handling/retry/recovery strategy is implemented (beyond task failure stop).
8) Execution mode ambiguity for decision-time device statuses:
   - When `UGO_SIMULATE_DEVICES=1`, statuses are synthetic from `SimulatedRobotClient`.
   - Real PACKML-based decisions (e.g., via `UpdateWorldState_From_uLM`) require live mode (`UGO_SIMULATE_DEVICES=0`) and reachable backend task execution.
